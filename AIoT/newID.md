# แนวทางย้าย GitHub Copilot ID แล้วพัฒนา Project AIoT ต่อเนื่อง

เอกสารนี้ออกแบบเพื่อให้ย้ายบัญชี Copilot แล้วยังพัฒนาโปรเจกต์ต่อได้ทันที โดยให้ Agent ทบทวนงานจาก spec.md อย่างเป็นระบบ

## 1) เป้าหมาย

- ย้ายไปใช้ Copilot ID ใหม่โดยไม่เสียบริบทงาน
- ให้ Agent เข้าใจ requirement ปัจจุบันจาก spec.md และสถานะระบบจริง
- ลดเวลาปรับตัวรอบแรกหลังย้าย ID

## 2) สิ่งที่ต้องเตรียมก่อนย้าย ID

1. Push งานโค้ดล่าสุดขึ้น Git ให้ครบทุกไฟล์ที่ต้องการเก็บ
2. ยืนยันว่าไฟล์เอกสารหลักอัปเดตล่าสุดแล้ว
   - spec.md
   - README.md
   - deploy/deploy.sh (ถ้ามีการแก้ flow)
3. เก็บ environment สำคัญไว้ในเครื่องปลายทาง
   - .env.local
   - ค่าการเชื่อม MQTT
4. บันทึกสถานะ deploy ล่าสุด (ผ่าน/ไม่ผ่าน, เวลา, branch)

## 3) ขั้นตอนหลังย้าย ID (ครั้งแรก)

1. Clone หรือเปิดโปรเจกต์เดิม
2. ติดตั้ง dependency
3. Build และรันระบบให้ผ่านก่อนเริ่มคุยกับ Agent
4. เปิด spec.md ให้เป็นไฟล์อ้างอิงหลัก

คำสั่งแนะนำ:

```bash
cd myIoTM3/AIoT
npm install
npm run lint
npm run build
```

## 4) Prompt ตั้งต้นที่แนะนำให้ส่ง Agent (คัดลอกใช้ได้เลย)

```text
ช่วยทบทวนโปรเจกต์จากไฟล์ spec.md ก่อน แล้วสรุป:
1) requirement ที่เป็น Must-have ปัจจุบัน
2) จุดเสี่ยง production ที่ต้องเช็กทุกครั้งก่อน deploy
3) รายการทดสอบสั้นๆ ที่ต้องรันหลังแก้โค้ด
จากนั้นให้เสนอแผนงานถัดไปที่มี impact สูงสุด 3 ข้อ
```

## 5) Prompt สำหรับเริ่มลงมือทำงานต่อ

```text
ให้ยึด spec.md เป็น source of truth
- ถ้ามีโค้ดที่ไม่ตรง spec ให้เสนอ patch ที่เล็กที่สุด
- หลังแก้โค้ดให้รัน lint/build
- ถ้ามี endpoint ใหม่ ให้ใส่วิธีทดสอบด้วย curl
- สรุปผลเป็นรายการ: changed files, behavior change, verification
```

## 6) รูปแบบการทำงานร่วมกับ Agent (แนะนำ)

1. เริ่มทุกงานด้วยขอบเขตงานที่ชัดเจน 1-3 ข้อ
2. บังคับให้ Agent อ้างอิง spec.md ก่อนแก้จริง
3. ให้ Agentตรวจครบ 3 ส่วนทุกครั้ง
   - code correctness
   - runtime behavior
   - deployment impact
4. ปิดงานด้วยหลักฐานยืนยัน
   - lint/build ผ่าน
   - API check ผ่าน
   - หน้าจอที่เกี่ยวข้องทำงานจริง

## 7) Checklist ก่อน merge/deploy

- [ ] งานตรงกับ spec.md ข้อที่เกี่ยวข้อง
- [ ] ไม่มี fallback/demo behavior ที่หลุดเข้า production
- [ ] lint ผ่าน
- [ ] build ผ่าน
- [ ] endpoint ที่แก้มี test ด้วย curl หรือ browser check
- [ ] บันทึกสิ่งที่เปลี่ยนในเอกสาร (README/spec.md ตามความเหมาะสม)

## 8) กรณี Agent หลง requirement ให้ดึงกลับอย่างเร็ว

ใช้ prompt นี้:

```text
หยุดทำ patch เพิ่มก่อน
ช่วยเทียบโค้ดปัจจุบันกับ spec.md เฉพาะหัวข้อที่เกี่ยวข้องกับงานนี้
สรุปเป็น: mismatch, risk, patch ที่ควรทำ (เล็กที่สุด)
แล้วค่อยลงมือแก้
```

## 9) เอกสารขั้นต่ำที่ควรคงไว้เสมอ

- spec.md: requirement ปัจจุบัน
- README.md: วิธีรันระบบและ deploy
- deploy notes: ปัญหา production ที่เคยเจอ + วิธีแก้

## 10) แนวคิดสำคัญ

เมื่อย้าย Copilot ID ให้คิดว่า "บริบทในแชตหายได้ แต่บริบทในไฟล์ต้องไม่หาย" 
ดังนั้นให้ลงทุนกับ spec.md และเอกสารการ deploy ให้ชัดเจน แล้วใช้ Agent เป็นตัวเร่งการพัฒนา ไม่ใช่เป็นที่เก็บความจำหลักของโปรเจกต์
